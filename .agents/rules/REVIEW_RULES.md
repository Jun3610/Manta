# MANTA2 프로젝트 코드 리뷰 및 검증 가이드라인

## 1. 역할 분담 (Multi-Agent Task)
- **개발 에이전트(Coder):** 기능 구현 및 파일 수정을 담당한다.
- **QA 에이전트(Tester):** 수정된 코드가 정상 구동되는지 백그라운드 터미널을 통해 검증한다.
- **리뷰 에이전트(Reviewer):** 작성된 코드의 예외 처리, 중복 로직, 보안 취약점을 엄격히 상호 검증(Cross-review)한다.

## 2. 코드 가독성 및 컨벤션 룰
- **PEP 8 준수:** 모든 파이썬 코드는 PEP 8 스타일 가이드를 철저히 준수해야 한다.
- **비동기 예외 처리:** `async/await` 함수 내부에는 반드시 적절한 `try-except` 예외 처리 블록이 포함되어 있는지 감시한다.
- **중복 리스너 금지:** 디스코드 봇 이벤트(`on_message` 등)가 중복 등록되어 동일 응답이 두 번 나가는 설계 결함이 없는지 반드시 상호 검수한다.

## 3. 출력 제약 조건
- 답변을 작성할 때 절대 한자나 중국어 템플릿을 노출하지 않는다. 무조건 100% 매끄러운 한국어만 사용한다.

## 4. Git 워크플로우 (필수 — 모든 작업에 적용)

### 브랜치 전략
- 새 기능/버그수정 작업 시작 시 반드시 아래 형식으로 브랜치를 생성한다:
  ```
  feat/phase0-[기능명]
  예: feat/phase0-rollback, feat/phase0-memory, feat/phase0-bulk-fix
  ```
- 브랜치는 항상 최신 `main`에서 분기한다: `git checkout -b feat/phase0-[기능명]`

### 작업 완료 후 필수 순서
작업이 완료되면 **반드시 아래 순서를 직접 실행**한다.

1. **보고서 저장**: `reports/YYYY-MM-DD_작업명.md` 생성 (첫 줄 반드시 `# 제목`)

> ⚠️ git add / commit / push는 사용자 권한이다. 에이전트가 직접 실행하지 않는다.

### 커밋 메시지 규칙 (참고용 — 사용자가 직접 커밋 시 사용)
```
feat: 새 기능 추가
fix:  버그 수정
refactor: 리팩토링 (기능 변경 없음)
docs: 문서/SPEC.md 업데이트
chore: 설정 변경 (config, gitignore 등)
```

### 주의사항
- `main` 브랜치에 직접 커밋하지 않는다.
- 보고서 없이 커밋하지 않는다.
